# sp-builder — генератор интерактивных HTML из СП-markdown

Обобщённый конвертер корпуса `normative-md/` (323 СП, Docling) в самодостаточные
интерактивные HTML-страницы в дизайн-токенах Komplid (oklch, светлая/тёмная тема).
Идейная база — [plannotator/effective-html](https://github.com/plannotator/effective-html):
один файл, ноль внешних запросов. Планы и стандарты: `../docs/01-SP-PAGES-PLAN.md`.

Стандартная библиотека Python 3.11+. Единственная опциональная зависимость —
`pymupdf` для восстановления картинок из исходных PDF (`pip install pymupdf`).
ИИ в парсинге не используется — только детерминированные правила и регулярные выражения.

## Использование

```bash
# ВСЕ документы из папки → HTML в папку, одной командой
python3 sp-builder/build.py --all -i normative-md -o output
# по умолчанию: -i input/ (если существует, иначе normative-md/), -o output/
# итог: output/*.html + output/img/<slug>/* + output/build-report.json

# один документ → sp-html/<slug>.html
python3 sp-builder/build.py "normative-md/SP_48.13330.2019_«СНиП 12-01-2004 Организация строительства».md"

# картинки восстанавливаются из downloads_faufcc/ автоматически (имена PDF = имена md);
# другую папку с PDF можно указать явно:
python3 sp-builder/build.py --all --pdf-dir downloads_faufcc

# дополнительно выгрузить промежуточную JSON-модель (контракт для Next.js-интеграции)
python3 sp-builder/build.py "<файл>" --json

# показать строки, потерянные при конвертации (автопроверка полноты)
python3 sp-builder/build.py "<файл>" --verbose

# отказ при плохих метриках качества (формулы/картинки/полнота) — гейт публикации
python3 sp-builder/build.py "<файл>" --strict

# обновить реестр корпуса (registry.json) по именам файлов normative-md/
python3 sp-builder/build.py --scan
```

## Гарантии полноты

После каждой сборки выполняется автопроверка: каждая содержательная строка исходного
markdown (включая ячейки таблиц) ищется в тексте итогового HTML; не найденное — метрика
`lostLines` (0 у эталона). Легитимные пропуски (строки «Содержания» с точками-лидерами,
колонтитулы, повторы титула) занесены в whitelist. Известные потери самого корпуса
(не генератора): формулы и картинки выброшены Docling'ом на этапе PDF→md — картинки
восстанавливаются из PDF (`--pdf-dir`), нераспознанные формулы помечаются честной
плашкой со ссылкой на официальное издание.

## Что умеет страница

- липкое оглавление со scroll-spy (на мобильном — выдвижная панель); **сворачиваемые
  ветки у каждого раздела**: подразделы, пункты с превью текста и термины — шеврон
  раскрывает, активная ветка раскрывается сама при чтении;
- картинки, восстановленные из исходного PDF (`img/<slug>/fig-N.png`, вне HTML — бюджет размера);
- поиск по пунктам и терминам с упрощённой морфологией («скрытые работы» находит «скрытых работ»);
- **стабильный якорь на каждый пункт** (`#p-8-2-3`) + кнопка «скопировать ссылку» — публичный
  контракт для цитирования (в т.ч. AI-ассистентами), схему НЕ менять;
- светлая/тёмная тема (localStorage `komplid-theme`), прогресс-бар чтения, print-CSS;
- глоссарий §3 как `<dl>` с источниками, приложения, библиография с якорями `#bib-N`;
- перекрёстные ссылки: `[N]` → библиография, «приложение Б» → якорь, «раздел 9» → якорь,
  упоминания других СП → ссылка, если документ опубликован в `registry.json`;
- SEO/AEO-слой: title/description/canonical из оверлея, JSON-LD TechArticle +
  BreadcrumbList + FAQPage, бейджи статуса, дисклеймер с ссылкой на официальный источник.

## Файлы

| Файл | Роль |
|---|---|
| `build.py` | CLI (один файл / `--all` батч / `--scan`), автопроверка полноты, метрики качества, бюджет размера (1.5/3 МБ), build-report.json |
| `cleanup.py` | чистка Docling-артефактов: хангыль-глифы, разрядка, номера страниц, `м 3 /с`→`м³/с` |
| `parser.py` | markdown → модель: метаданные из преамбулы, дерево разделов/пунктов, глоссарий, приложения (включая демоушен промоутнутых заголовков), библиография, image/formula-стабы |
| `renderer.py` | модель + оверлей + картинки → HTML (CSS/JS инлайном, без внешних ресурсов) |
| `pdf_images.py` | извлечение растров из исходных PDF (PyMuPDF), сопоставление со стабами по порядку |
| `registry.json` | реестр корпуса: designation → slug/файл/название; ручные поля (`published`) переживают `--scan` |
| `overrides/<slug>.json` | ручной оверлей: статус, TL;DR, FAQ, CTA, related, SEO — см. эталон `sp-48-13330-2019.json` |

## Правила

1. **Схема слагов и якорей — публичный контракт** (`sp-48-13330-2019`, `#p-8-2-3`,
   `#pril-b`, `#tbl-2`, `#term-3-12`, `#bib-26`). Менять только с redirect-слоем.
2. Текст норм не редактируется и не рерайтится — чистится только форма (глифы, переносы).
3. Публикация документа = оверлей заполнен (статус + дата сверки + TL;DR + FAQ) и
   метрики качества в норме (`--strict` проходит). Без оверлея генератор работает,
   но такие страницы — только для внутреннего просмотра.
4. Никаких внешних ресурсов в HTML (шрифты — системные стеки).
