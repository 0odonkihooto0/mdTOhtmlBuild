# 01 — План внедрения СП-страниц (md → интерактивный HTML)

> Вход: `normative-md/*.md` (323 СП, Docling-конвертация из PDF).
> Инструмент: `sp-builder/` (этот репо) — обобщённый Python-генератор.
> Выход: самодостаточные HTML в `sp-html/` → далее в `komplid-marketing`.
> Эталон: **СП 48.13330.2019 «Организация строительства»**.
> Стратегия и blindspot-реестр: `00-STRATEGY-SEO-AEO-GEO.md`.

---

## 1. Почему СП 48 — эталон

- Профильный для Komplid: п. 8.2 «Исполнительная документация», п. 8.3 «Порядок освидетельствования
  скрытых работ», раздел 9 «Строительный контроль», Приложение Б «Примерный состав исполнительной
  документации» — прямые мостики к /pto, /shablony/aosr, статьям блога.
- Технически чистый: 2 779 строк, **0 потерянных формул**, 1 потерянный рисунок, таблицы на месте.
- Средний размер → отработка всех механик без chunking-а.

## 2. Архитектура генератора `sp-builder/`

```
sp-builder/
├── build.py            # CLI: python sp-builder/build.py "normative-md/SP_48....md" -o sp-html/
├── parser.py           # markdown → структурная модель документа (JSON-совместимая)
├── renderer.py         # модель + оверлей → самодостаточный HTML (тема Komplid oklch)
├── cleanup.py          # чистка Docling-артефактов (хангыль-глифы, разрядка, переносы)
├── registry.json       # реестр корпуса: designation → slug, файл, статус, метрики качества
└── overrides/
    └── sp-48-13330-2019.json   # ручной оверлей эталона (TL;DR, FAQ, статус, CTA)
```

Только стандартная библиотека Python 3.11 — без pip-зависимостей (парсим markdown, не PDF).

### 2.1. Пайплайн парсера (стадии)

1. **Чистка** (`cleanup.py`): хангыль→латиница/греческий (таблица перекодировки из
   `pdf-to-html/pdf_extract.py`), разрядка «Т а б л и ц а»→«Таблица», склейка переносов,
   нормализация `м 3 /с`→`м³/с` строго после единиц измерения (без изменения числовых значений!).
0. **Картинки** (`pdf_images.py`): Docling выбросил рисунки (`<!-- image -->`) — растры
   извлекаются из исходных PDF в `downloads_faufcc/` (имена совпадают с md 1:1, PyMuPDF)
   и сопоставляются со стабами по порядку; нераспознанное — честная плашка со ссылкой
   на официальное издание. Автопроверка полноты (lostLines) — на каждой сборке.
2. **Преамбула**: всё до первого `## N <Название>` — извлекаем метаданные регулярками:
   обозначение (`СП 48.13330.2019`), СНиП-предок, название, «УТВЕРЖДЕН приказом … № 861/пр»,
   «введен в действие с …». Остальную преамбулу (исполнители, предисловие) сворачиваем
   в закрываемый блок «О документе».
3. **Оглавление-мусор**: секцию `## Содержание` (вырожденная таблица с точками-лидерами)
   выбрасываем — TOC строим сами из фактической структуры.
4. **Дерево разделов**: `## N Название` → раздел; `## N.M Название` → подраздел;
   вырожденные заголовки (`## 3.53` без текста, строки приложений, промоутнутые ячейки таблиц) —
   эвристики понижения до текста/термина.
5. **Пункты**: внутри раздела абзацы, начинающиеся с `N.M[.K]` → пункт с ID `p-N-M-K`.
   Пункт — минимальная адресуемая единица (якорь + кнопка копирования ссылки).
6. **Спец-секции**: §2 «Нормативные ссылки» → список ссылок; §3 «Термины и определения» →
   глоссарий `<dl>` (термин + определение + номер); «Приложение А/Б/…» → приложения
   (ID `pril-a`, пункты `pril-b-1`); «Библиография» → нумерованный список.
7. **Таблицы**: GFM-таблицы → семантический `<table>` с `<caption>` («Таблица N — Название»,
   caption ищем в предшествующей строке). ID `tbl-N`.
8. **Инлайн-обогащение**: упоминания `СП NNN.NNNNN`/`ГОСТ …` → если документ есть в
   `registry.json` — ссылка `<a href="sp-….html">`, иначе `<span class="std">`;
   `[N]` → ссылка на библиографию; «таблице N» → ссылка на `#tbl-N`.
9. **Метрики качества** (blindspot B-1): counts formula-stub / image-stub / битых таблиц →
   в `registry.json`; при `--strict` документы ниже порога не собираются.

### 2.2. Модель данных (промежуточный JSON)

```jsonc
{
  "meta": {
    "designation": "СП 48.13330.2019", "slug": "sp-48-13330-2019",
    "title": "Организация строительства", "snip": "СНиП 12-01-2004",
    "approval": {"order": "№ 861/пр", "date": "2019-12-24", "effective": "2020-06-25"},
    "quality": {"formulaStubs": 0, "imageStubs": 1, "tables": N}
  },
  "sections": [{ "num": "8", "title": "…", "id": "sec-8",
    "children": [{ "num": "8.2", "title": "Исполнительная документация", "id": "sec-8-2",
      "clauses": [{ "num": "8.2.1", "id": "p-8-2-1", "html": "…" }] }] }],
  "glossary": [{ "num": "3.1", "term": "…", "def": "…" }],
  "normRefs": ["СП 20.13330…", …],
  "appendices": [{ "letter": "Б", "title": "…", "id": "pril-b", "blocks": [...] }],
  "biblio": ["…"]
}
```

Эта модель — **контракт на оба выхода**: standalone HTML сейчас и Next.js-рендер потом (§7).

### 2.3. Оверлей (ручная добавленная ценность)

`overrides/<slug>.json` — необязателен (генератор деградирует изящно), для публикуемых
документов обязателен по чек-листу:

```jsonc
{
  "status": { "state": "active",            // active | amended | superseded
              "note": "Действует. Применяется на добровольной основе (не входит в перечень по ПП РФ № 815), если иное не установлено договором/заданием.",
              "checkedAt": "2026-07-10",
              "supersedes": "СП 48.13330.2011" },
  "tldr": "2–4 предложения: что регулирует документ, кому обязателен, что самое важное.",
  "keyHighlights": [ {"anchor": "sec-8-2", "label": "Состав и ведение исполнительной документации"} ],
  "faq": [ {"q": "…?", "a": "…"} ],        // ≥ 6 вопросов, попадают в FAQPage schema
  "cta": [ {"afterAnchor": "sec-8-2", "text": "…", "href": "https://komplid.ru/pto?utm_…"} ],
  "seo": { "title": "…", "description": "…", "canonical": "https://komplid.ru/normativ/sp-48-13330-2019" }
}
```

## 3. Схема слагов и якорей — ПУБЛИЧНЫЙ КОНТРАКТ (не менять!)

- Документ: `sp-<номер с дефисами>-<год>` → `sp-48-13330-2019`. ASCII, лower-case.
- Раздел: `#sec-8`, подраздел `#sec-8-2`.
- Пункт: `#p-8-2-3` (точки номера → дефисы).
- Приложение: `#pril-b`, пункт приложения `#pril-b-1`.
- Таблица: `#tbl-2`, `#tbl-b-1`. Термин: `#term-3-12`.

Смена схемы после публикации ломает внешние ссылки и AI-цитаты (blindspot C-3) —
любое изменение только с redirect-слоем.

## 4. Структура страницы (сверху вниз)

1. **Шапка**: хлебные крошки → обозначение + название; бейджи: статус (цвет по state),
   «Приказ № 861/пр от 24.12.2019», «Введён 25.06.2020», дата сверки.
2. **Плашка-дисклеймер**: «Справочная копия. Официальное опубликование — minstroyrf.gov.ru» (ссылка).
3. **«Главное» (TL;DR)** из оверлея + ключевые разделы (keyHighlights) — AEO-магнит.
4. **Липкое оглавление** (левая колонка, scroll-spy; на мобильном — выезжающее).
5. **Поиск по документу** (клиентский: индекс пунктов встроен в страницу, фильтр + подсветка + переход).
6. **Тело документа**: разделы/пункты с якорями и кнопкой «¶ скопировать ссылку», семантические
   таблицы, глоссарий как `<dl>`, приложения, библиография. Контекстные CTA из оверлея.
7. **FAQ** (из оверлея) + FAQPage schema.
8. **Связанные материалы**: другие СП (из «Нормативных ссылок», только имеющиеся в корпусе),
   шаблоны/калькуляторы/статьи komplid.ru (из оверлея).
9. **Подвал**: дисклеймер повторно, «© текст норматива — Минстрой России».

UI-слой: тема Komplid oklch (токены из `pdf-to-html/html_gen.py` ≡ `design/landing-reference.html`),
light/dark с сохранением в localStorage, прогресс-бар чтения, print CSS (оглавление и кнопки
скрываются, якоря печатаются как номера пунктов). Никаких внешних ресурсов (blindspot D-4).

## 5. SEO/AEO/GEO-слой страницы

- `<title>`: «СП 48.13330.2019 «Организация строительства» — актуальный текст с удобной навигацией».
- JSON-LD: `TechArticle` (headline, datePublished=дата приказа, dateModified=дата сверки, isBasedOn=официальный источник) + `BreadcrumbList` + `FAQPage`.
- H2-структура тела = разделы СП; FAQ даёт вопросные H3.
- `<html lang="ru">`, canonical на будущий URL komplid.ru (из оверлея).
- Каждый пункт — `<section id>` с микро-заголовком: LLM-краулеры получают адресуемые чанки (GEO).
- Позже, на komplid.ru: включение в сегментный sitemap-normativ.xml и llms.txt.

## 6. Chunking больших СП (волна 2+, не для эталона)

Порог: если HTML > 1.5 МБ или > ~5 000 строк markdown → режим «сборник»:
- `/normativ/<slug>` — оглавление + метаданные + TL;DR + FAQ (это главная SEO-страница);
- `/normativ/<slug>/razdel-<N>` — страницы разделов, `isPartOf` в schema.org, перелинковка prev/next;
- якоря пунктов сохраняют формат `#p-…` внутри своей страницы раздела; `registry.json` хранит карту
  «пункт → страница», чтобы кросс-ссылки из других СП вели сразу на нужный chunk.

## 7. Интеграция в komplid-marketing (двухэтапно)

**Этап 1 — быстрый (standalone):** файлы `sp-html/*.html` кладутся в
`komplid-marketing/public/normativ/<slug>.html`. Плюсы: ноль кода в Next.js, мгновенный запуск.
Минусы: нет общего header/footer. Приемлемо для проверки спроса (волна 1).

**Этап 2 — нативный:** `sp-builder` уже выдаёт промежуточный JSON (§2.2) → в komplid-marketing
добавляется `src/app/normativ/[slug]/page.tsx` + `src/lib/normativ-data.ts` (по образцу
`kalkulyator`), рендерящий ту же модель в общем layout. Standalone-URL получают 301 на нативные.
Канонический URL с первого дня указывает на нативный путь `/normativ/<slug>` — переезд не ломает SEO.

Хаб `/normativ` (каталог по категориям: организация строительства, конструкции, инженерка,
пожарная безопасность…) — из `registry.json`.
